# 学习总结：网络请求中的 3 个问题与优化思路

## 问题 1：不能在主线程请求网络

### 现象

如果直接在主线程发起网络请求，页面会卡住。请求耗时较长时，还可能触发 ANR。

### 原因

Android 的 UI 绘制、点击响应都运行在主线程。网络请求、文件读写、图片解码都属于耗时操作，放在主线程会阻塞界面刷新。

### 本项目做法

`ImageRepository` 使用线程池执行网络请求和磁盘缓存读写，请求完成后通过主线程 `Handler` 回调 Activity 更新 UI。

### 优化思路

后续可以使用 Kotlin 协程，把 `Dispatchers.IO` 和 `Dispatchers.Main` 结合起来，让异步代码更清晰。

## 问题 2：重复下载浪费流量和时间

### 现象

如果每次打开页面都重新请求图片，加载速度慢，也会浪费用户流量。

### 原因

图片资源通常体积较大，而且同一个 URL 对应的内容在短时间内不会频繁变化。重复下载没有必要。

### 本项目做法

使用多级缓存：

- 内存缓存：当前进程内快速复用 Bitmap。
- 图片文件缓存：应用重启后仍然可以复用已经下载过的图片文件。
- OkHttp 响应缓存：缓存 HTTP 原始响应，在图片文件缓存被清除时仍可避免真实网络请求。

图片文件按 `SHA-256(imageUrl)` 命名，URL 不变时缓存 key 不变。下次启动时，如果磁盘文件存在，就直接解码本地文件，不再发起网络请求。

### 优化思路

后续可以增加缓存过期时间和最大缓存容量，避免缓存文件无限增长。

## 问题 3：弱网或异常响应会影响体验

### 现象

网络断开、服务端返回错误、响应体为空、图片文件损坏，都可能导致图片加载失败。

### 原因

移动网络环境不稳定，图片请求链路中任何一个环节失败，都会影响最终展示。

### 本项目做法

- OkHttp 设置连接超时和读取超时。
- HTTP 状态码非成功时直接进入错误回调。
- 响应体为空时抛出异常。
- 图片解码失败时抛出异常。
- UI 层显示「加载失败」并保留页面结构。

### 优化思路

后续可以加入：

- 自动重试。
- 失败占位图。
- 离线提示。
- 网络恢复后重新加载。
- 根据实际图片尺寸进行压缩解码，降低内存压力。
