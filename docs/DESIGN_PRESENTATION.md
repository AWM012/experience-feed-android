# 方案设计与思考过程

这份文档用于作业点评环节，可以作为 3 分钟左右介绍稿的提纲。

## 1. 我做了什么

这次作业我实现了一个参考抖音「经验」栏形态的 Android 图文浏览应用，包含两个核心页面：

- 首页：双列图片流列表。
- 详情页：大图预览和图文信息展示。

每张图片都来自网络请求，图片加载后会写入本地缓存。再次进入页面时，如果图片已经下载过，会优先读取内存缓存或磁盘缓存，避免重复请求网络。

## 2. 为什么这样拆分

我把项目拆成了三层：

- `MainActivity` 和 `DetailActivity` 负责页面展示。
- `ImageRepository` 负责图片请求、缓存和解码。
- `FeedData`、`ImagePost` 和 `UiKit` 负责数据与通用工具。

这样拆分的原因是：Activity 只负责展示和交互，图片如何请求、如何缓存、如何判断来源都收敛到 `ImageRepository`。后续如果要把列表换成 RecyclerView，或者把图片库换成 Glide，页面层的改动会更小。

## 3. 首页双列流怎么实现

首页没有直接使用第三方瀑布流库，而是用 `ScrollView + LinearLayout` 手动实现。页面中有左右两列，每次添加卡片时，会根据当前左右两列的预估高度，把新卡片放到更短的一列。

这个方案的优点是：

- 代码直观，适合课程作业讲解。
- 可以清楚展示双列图片流的布局思路。
- 不依赖复杂组件，方便把重点放在网络请求和缓存上。

它的不足也很明确：如果数据量很大，应该使用 `RecyclerView + StaggeredGridLayoutManager`，因为 RecyclerView 有视图复用能力，性能更好。

## 4. 图片加载怎么设计

图片加载统一走 `ImageRepository.load()`，网络层使用 Retrofit + OkHttp：

1. 先查内存缓存。
2. 内存没有，再查磁盘缓存。
3. 图片文件也没有，再通过 Retrofit 调用下载接口，由 OkHttp 执行网络请求。
4. 下载成功后写入磁盘缓存。
5. 解码成 Bitmap。
6. 回到主线程更新 UI。

我没有在主线程做网络请求和文件读写，因为这些操作都可能阻塞界面，严重时会导致 ANR。项目中使用线程池处理耗时任务，再用主线程 Handler 回调页面。

## 5. 缓存为什么这样做

缓存分为三层：

- 内存缓存解决当前进程内重复加载的问题。
- 图片文件缓存解决应用重启后重复下载的问题。
- OkHttp 响应缓存保存 HTTP 原始响应，作为网络层缓存。

OkHttp 还配置了拦截器：请求头拦截器统一添加公共请求头，日志拦截器记录 URL、状态码和耗时，响应缓存拦截器统一设置缓存策略。

磁盘缓存文件名使用图片 URL 的 SHA-256 值。这样 URL 相同，缓存文件名就稳定；同时也避免 URL 中的特殊字符直接出现在文件名里。

为了让演示更清楚，我在首页和详情页都显示了加载来源：网络下载、磁盘缓存、内存缓存。这样录屏时能直观看到缓存是否生效。

## 6. 详情页做了哪些处理

详情页展示大图、标题、描述、日期、拍摄位置、图片大小、缓存路径和加载来源。

我额外加了「重新下载图片」按钮，用来演示绕过缓存后重新请求网络；还增加了「测试 OkHttp 响应缓存」按钮，用于展示 HTTP 缓存命中。

另外，模拟器使用新版本 Android 时，内容可能会延伸到状态栏和导航栏区域，所以我处理了顶部和底部安全区域，避免返回按钮贴到状态栏，底部按钮被导航条遮挡。

## 7. 遇到的问题和解决方式

第一个问题是网络请求不能放在主线程。解决方式是把网络请求、文件读写、图片解码放到后台线程池。

第二个问题是图片重复下载会浪费时间和流量。解决方式是实现内存缓存和磁盘缓存，并在页面上显示加载来源。

第三个问题是新系统的状态栏区域会影响布局。解决方式是在 UI 工具类里获取状态栏和导航栏高度，页面布局主动留出安全距离。

## 8. 我的取舍

这次我没有使用 Glide、Coil 或 RecyclerView，主要是为了让作业重点更清楚：自己实现 OkHttp 请求、缓存复用和双列布局。

如果这是线上项目，我会优先换成：

- `RecyclerView + StaggeredGridLayoutManager`
- 成熟图片库负责图片生命周期、采样压缩和缓存清理
- ViewModel 管理页面状态
- 分页加载和失败重试

但作为课程作业，现在这个实现更适合展示基础能力和方案思考。

## 9. 点评时可以这样总结

我这次的方案重点是把图片流、详情页、网络请求和缓存链路完整串起来。页面层只处理展示，图片加载和缓存统一放在仓库层。首页通过左右列高度估算实现轻量瀑布流，详情页复用同一套图片仓库，所以可以自然命中缓存。这个实现不是最复杂的工程化方案，但它把作业要求中的 OkHttp 请求、双列列表、大图预览、图片描述和历史缓存都覆盖到了。
